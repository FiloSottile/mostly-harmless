Static CT partial tile read load

A Static CT API log serves level 0 entries in tiles of 256. While a tile is incomplete, a tailing client must re-fetch it as a partial tile on every poll, re-downloading the entries it already has. For a slow-growing log this wastes a lot of bandwidth: a log growing 10 leaves/s polled every second yields partials of 10, 20, 30, …, 250, then 256+4, 14, … This page shows the resulting steady-state read rate measured in leaves/second.

A client that can tolerate some latency can avoid partial tile refetches by fetching only newly-completed full tiles, and only falling back to the trailing partial when there are none. The skip slider below models this torchwood-style policy:

  • N = 0 — naïve baseline: fetch everything up to the log tip on every poll.
  • N = 1 — torchwood exactly: fetch only full tiles when any have completed since the cursor, otherwise fall back to the partial immediately, with no waiting.
  • N ≥ 2 — torchwood plus patience: wait N − 1 polls between partial fetches on the partial-only branch. Completed full tiles still trigger an immediate fetch.

The rate hits the ideal as soon as growth reaches 256 / N leaves per poll — at or above that threshold, a new full tile always completes before the patience runs out and the partial is never fetched. So N = 1 bottoms out at one tile per poll, N = 2 at half a tile, N = 4 at a quarter, N = 8 at an eighth.
