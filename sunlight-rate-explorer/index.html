Static CT partial-tile read load

A Static CT API log serves level-0 entries in tiles of 256. While a tile is incomplete, a tailing client must re-fetch it as a partial tile on every poll, re-downloading the entries it already has. For a slow-growing log this wastes a lot of bandwidth: a log growing 10 leaves/s polled every second yields partials of 10, 20, 30, …, 250, then 256+4, 14, … This page shows the resulting steady-state read rate measured in leaves/second.

A client that can tolerate some latency can avoid partial-tile refetches by fetching only newly-completed full tiles, and only falling back to the trailing partial after waiting a few polls. The skip slider below models this torchwood-style policy: at N = 0 the client naïvely fetches everything up to the log tip on every poll; at N ≥ 1 it fetches only full tiles when any have completed since the last fetch, and falls back to the partial only after N polls of no progress. Whenever each poll crosses a tile boundary (growth ≥ ~one tile/poll) partials are never fetched and the rate equals the ideal.
